## 简介
本文件聚焦于“进度监控端点”的详细API文档，涵盖以下两个端点：
- GET /api/status/{job_id}：基于 Server-Sent Events（SSE）的实时状态流
- GET /api/status/{job_id}/json：返回当前转换状态的JSON响应（用于非SSE客户端的降级）

文档将深入说明：
- SSE 实现原理与事件格式
- 连接管理与生命周期控制
- 转换状态枚举（ConversionStage）与进度百分比计算
- 实时状态推送机制
- SSE 客户端实现示例与JSON端点使用方法
- 错误处理策略、浏览器兼容性与降级方案

## 详细组件分析

### SSE 端点：GET /api/status/{job_id}
- 功能概述
  - 以 Server-Sent Events 形式持续推送转换状态，直到达到完成或错误阶段
  - 事件格式为纯文本事件流，每条事件包含一个JSON对象（序列化后的 ConversionStatus）
- 实现要点
  - 事件生成器在循环中读取当前状态并yield事件
  - 当状态为 complete 或 error 时停止推送
  - 设置合适的HTTP头以禁用缓存、保持连接、避免代理缓冲
- 事件格式
  - 每次推送一条 data: 行，后跟双换行作为事件结束
  - data 内容为 ConversionStatus 的JSON字符串
- 连接管理
  - 客户端应正确处理连接中断与重连
  - 服务器端在作业不存在时会主动终止流

```mermaid
flowchart TD
Start(["进入 /api/status/{job_id}"]) --> CheckJob["检查 job_id 是否存在"]
CheckJob --> Exists{"存在？"}
Exists --> |否| Close["关闭连接404"]
Exists --> |是| Loop["循环读取状态并推送"]
Loop --> GetStatus["获取当前状态"]
GetStatus --> Serialize["序列化为 JSON 字符串"]
Serialize --> Yield["yield data: JSON + 双换行"]
Yield --> DoneCheck{"stage 是否 complete 或 error？"}
DoneCheck --> |是| Stop["停止推送并关闭连接"]
DoneCheck --> |否| Sleep["等待 1 秒"]
Sleep --> Loop
```

### 转换状态与进度计算
- 状态枚举（ConversionStage）
  - uploaded：文件已上传
  - parsing：解析文件内容
  - splitting：检测章节
  - extracting_characters：提取角色
  - converting：逐章转换
  - assembling：组装剧本
  - validating：验证输出
  - complete：完成
  - error：发生错误
- 进度百分比（progress_percent）
  - 在不同阶段按固定权重推进
  - 在 converting 阶段，根据 total_chapters 与 current_chapter 动态计算
  - 示例计算逻辑：30 + (50 × 当前章节/总章节)
- 章节信息
  - total_chapters：检测到的章节总数
  - current_chapter：当前正在转换的章节号

### JSON 端点使用方法
- 客户端轮询策略
  - 每隔固定时间（如1.5秒）请求一次 /api/status/{job_id}/json
  - 根据 stage 判断是否继续轮询
  - complete 时停止轮询并处理结果
  - error 时停止轮询并提示错误
- 适用场景
  - 无法使用SSE的环境（如某些代理/CDN）
  - 简单的非浏览器客户端

### 错误处理策略
- 作业不存在
  - SSE：连接建立后若作业不存在，服务器会主动终止流
  - JSON：返回404，客户端应停止轮询并提示错误
- 转换过程中异常
  - 后台任务捕获异常并设置 stage 为 error，同时记录 error_message
  - SSE：在最后一次事件后终止流
  - JSON：轮询返回 error 状态，客户端显示错误信息
- 重复启动转换
  - 若作业已在 converting 阶段，再次启动会返回400

## 性能考量
- SSE 连接
  - 服务器端每秒推送一次，适合低延迟实时反馈
  - 注意代理/网关对长连接与缓冲的限制
- 轮询策略
  - 建议轮询间隔适中（如1.5秒），避免过度请求
  - 在 complete/error 后及时停止轮询
- 状态更新频率
  - 转换过程中的状态更新由后台任务驱动，SSE/JSON端点仅读取现有状态
  - 避免在客户端频繁发起请求导致不必要的负载

## 故障排查指南
- 无法接收SSE事件
  - 检查网络代理/CDN是否支持SSE
  - 确认服务器端已设置正确的响应头（Cache-Control、Connection、X-Accel-Buffering）
  - 在浏览器开发者工具中查看网络面板的事件流
- 轮询返回404
  - 确认 job_id 是否正确且作业尚未过期
  - 检查作业是否存在（_jobs字典）
- 进度停滞
  - 检查后台转换任务是否仍在运行
  - 查看日志中是否有异常并导致状态停留在某个阶段
- 错误状态
  - 查看 error_message 字段了解具体错误原因
  - 重新尝试转换或修正输入

## 结论
- SSE 端点提供低延迟、高效的实时进度反馈；JSON 端点为非SSE环境提供可靠降级方案
- 状态模型与枚举确保前后端一致的状态表达与进度计算
- 前端脚本展示了SSE与轮询两种实现路径，具备良好的兼容性与可维护性
- 建议在生产环境中结合代理/网关配置优化SSE传输，并为非SSE客户端提供稳定的轮询策略
